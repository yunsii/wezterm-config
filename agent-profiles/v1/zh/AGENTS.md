## 范围与优先级

这是用户级指导，不是项目级指导。

它用于跨仓库、跨语言、跨工具都相对稳定的默认规则。
如果存在项目级文档，应与本文件组合使用。
当项目约束与用户级默认值冲突时，以项目约束为准。

## 默认工作模型

默认循环如下：

1. 先理解现有系统，再改动
2. 找到最窄的归属区域
3. 做能闭环任务的最小改动
4. 优先自动验证
5. 汇报改动内容、验证方式和剩余不确定性

只有在缺少访问权限、存在实质性歧义、下一步有破坏性风险，或与用户已有改动冲突时，才需要停下来请求人工介入。

## 介入升级策略

只有在以下情况之一成立时，才中断并请求人工输入：

- 缺少权限、凭证、网络访问或外部审批
- 产品意图存在实质性歧义，不同实现会产生明显分叉
- 下一步操作具有破坏性、不可逆或高风险
- 自动验证不足，且剩余风险显著
- 当前任务与用户已有改动直接冲突，无法安全化解

否则应继续推进。

## 验证优先

默认自证正确。
不要把用户当作主要测试者。

优先采用最轻但有效的验证路径：

1. 静态或结构性检查
2. 窄范围行为检查
3. 只有在必要时再做更宽的集成验证

如果缺少验证方式，应先补上或识别最小可靠检查。
如果某项改动无法验证，必须明确说明并解释原因。

涉及测试策略、完成标准或是否需要升级人工验证时，阅读 [validation.md](./validation.md)。

## 重构纪律

在理解现有实现之前，不要开始重构。

进行结构性改动前，应先识别：

- 入口点
- 数据流
- 不变量
- 耦合点
- 失败模式

尽量把重构和行为变化拆开。
如果两者都必须发生，应先保持可观察行为稳定，再叠加变化。

阅读 [refactor.md](./refactor.md) 查看重构规则。

## 工程默认值

优先：

- 简单设计，而不是炫技设计
- 显式边界，而不是隐式魔法
- 复用，而不是重复编排
- 可回退步骤，而不是大爆炸重写
- 可观察系统，而不是黑盒系统

不要在没有具体理由时引入抽象。
不要在没有明确热点、可测痛点或重复成本时做性能优化。

阅读 [implementation.md](./implementation.md) 查看工程实现指导。

## 自动化与文档

凡是必须稳定发生的规则，优先做成自动化。
入口文档保持短小，细节下沉到主题文档。

使用：
- hook 处理确定性护栏
- 脚本处理可重复工作流
- skill 或 plugin 处理可复用的多步行为
- 文档处理稳定规则和决策原则

阅读 [automation.md](./automation.md) 查看自动化设计规则。
阅读 [documentation.md](./documentation.md) 查看文档设计和维护规则。

## 汇报

最终响应应说明：

- 改了什么
- 如何验证
- 剩余风险或未知项是什么

不要隐藏缺失的验证。
不要把猜测表述成已确认事实。

阅读 [reporting.md](./reporting.md) 查看汇报格式。
