# Implementation（中文对照）

Source of truth:
[../en/implementation.md](../en/implementation.md)

本文档是英文版的中文对照参考。
若中英文存在差异，以英文版为准。

## 何时阅读

当需要决定结构、抽象、模块边界、可靠性模式或性能权衡时。

## 何时不必阅读

当改动只是局部 bug 修复且无设计决策，或仅是纯文本/注释编辑。这些场景 AGENTS.md 的一句话原则即可。

## 默认值

优先选择易理解、可观察、比例合适、维护成本低的实现。

## 按归属组织

优先使用具备以下特征的结构：

- 清晰的入口点
- 显式边界
- 稳定接口
- 有限的隐藏耦合
- 少量关键行为归属点

没有强理由时，不要把一个职责拆散到很多文件里。

## 可靠性

优先选择具备以下特征的实现：

- 在可能时具备幂等性
- 显式表达失败
- 易于观察
- 易于重试
- 中途停止时安全
- 易于回滚

避免依赖隐藏前提的脆弱流程。

## 可维护性

优先：

- 少量但强的核心概念
- 窄接口
- 直接的控制流
- 只有在意图不明显时才写注释

避免：

- 投机性抽象
- 过度泛化
- 过深的隐式行为
- 没有稳定检查点的大重写

## 性能

不要默认做优化。

只有在满足以下至少一项时才优化：

- 存在已测量热点
- 在有意义的规模下存在重复成本
- 属于延迟敏感路径
- 存在显著影响体验的资源瓶颈

优化时，除非收益非常明确，否则应尽量保持可读性。

## 复用与重复

当现有机制确实合适时，优先复用。
如果复用会引入别扭抽象或隐藏耦合，就不要强行复用。

短小、显而易见的重复，通常比过早框架化更便宜。
当重复的是稳定语义下的编排流程时，更适合提取为 helper、script、skill 或 plugin。

## 改动精确性

只触碰任务需要改动的地方。

- 生成或编辑代码前，先把周围上下文读懂
- 沿用既有风格和模式，除非任务本身就是要改变它们
- 聚焦改动期间不顺手重构邻近代码、不重命名无关 identifier、不"整理周边"

## 改动规模

优先做小而闭环的改动。

如果任务较大，应拆成多个步骤，每一步都满足：

- 保持行为不变，或只清晰改变一件事
- 可以独立评审
- 可以独立验证
