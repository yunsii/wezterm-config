# Tool Use（中文对照）

Source of truth:
[../en/tool-use.md](../en/tool-use.md)

本文档是英文版的中文对照参考。
若中英文存在差异，以英文版为准。

## 何时阅读

当需要在多种工具之间选择、安排工具调用顺序，或判断是否需要调用工具时。

## 何时不必阅读

当任务只能用某个专用工具完成、没有其他现实选择时。

## 默认值

每次工具调用都有成本：占用用户注意力、触发权限提示、消耗 token 预算、增加延迟。
目标是减少调用次数，把独立工作批量化，并优先选择能回答当前问题的最窄工具。

## 专用工具优先于 shell

能用专用工具时，不要用 shell：

- 查看文件内容 → 读取工具，不用 `cat` / `head` / `tail`
- 内容检索 → grep 类工具，不用原生 `rg` / `grep`
- 路径发现 → glob 类工具，不用 `find` / `ls`
- 文件编辑 → 编辑/写入工具，不用 `sed` / `awk` / heredoc 重定向

shell 留给真正需要 shell 的动作：运行程序、查 git 状态、构建步骤、进程控制。

## 批量并行独立调用

多个没有相互依赖的工具调用，应在同一轮中一次性发起。
只有当下一步的输入依赖上一步输出时，才串行调用。

## 合并只读 shell 命令

如果必须用 shell，多条非交互式只读命令应该用 `&&` 合并成一次，让用户只看到一次确认而不是 N 次。

以下情况不合并：

- 带破坏性副作用的命令（每一条应可独立审阅）
- 中间输出还需要 agent 先查看再决定下一步的命令

## 写入前先读

在编辑或覆盖已有文件前，先把它读一次。
避免触发 harness 层的错误，也避免覆盖用户并发的改动。

## 任务追踪

当工作包含三个及以上独立步骤，或用户需要看到可见进度时，使用结构化任务追踪。
单步或纯对话场景不必使用。
开始步骤时标记为进行中，做完立即标记完成；不要把状态更新攒在一起。

## 子 agent

当工作量大且相互独立时（长篇研究、多路并行搜索、跨模块审查），委派给子 agent。
只有一两次直接工具调用就能解决时，不要委派 —— 给子 agent 讲清上下文的成本会超过节省。

## 失败处理

工具调用失败时，应该就地如实汇报，而不是静默重试。
不要在卡住的调用上继续消耗权限提示和上下文；先诊断，再调整。

## 宿主配置

宿主侧用于预批准命令或减少权限提示的机制（allowlist、设置文件、permission mode 等），属于宿主配置，不应放进本 profile。
请通过宿主平台自己的配置路径落地；本文件保持对具体宿主键名的零引用。
