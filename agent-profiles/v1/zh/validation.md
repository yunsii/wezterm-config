# Validation（中文对照）

Source of truth:
[../en/validation.md](../en/validation.md)

本文档是英文版的中文对照参考。
若中英文存在差异，以英文版为准。

当任务涉及正确性、完成标准、测试策略、上线信心，或是否需要让用户手动验证时，阅读本文件。

## 目标

默认目标是在尽量少打断人的前提下，以机器可验证的信心完成任务。

人工评审是兜底手段，不应成为首选验证策略。

## 完成标准

一个任务完成时，应同时满足以下条件：

- 请求的改动已经实现
- 相关系统仍按预期工作
- 已运行当前条件下最强且实际可行的自动验证
- 任何未验证风险都已被明确说明

## 验证顺序

优先使用最轻但有用的路径：

1. 先阅读并推断当前实现边界
2. 如果有静态验证，先运行静态验证
3. 运行能覆盖改动的最窄行为验证
4. 只有在改动规模确实需要时，再扩展到集成或端到端验证

如果更窄的检查已经足以建立正确性，不要直接跳到重型验证。

## 默认规则

如果 agent 能直接验证、脚本化验证，或继续缩小不确定性，就不要把手动验证抛给用户。

## 何时允许人工介入

只有在以下情况下，人工介入才是合理的：

- 需要访问权限或凭证
- 当前环境对 agent 不可用
- 结果依赖主观产品判断
- 涉及硬件、UI 或无法自动执行的外部系统
- 剩余不确定性已经无法继续缩小，只能由用户判断

## 缺少测试时

如果没有现成测试，不要自动停止。

在以下路径中选择其一：

- 如果仓库支持测试，就补一条最小聚焦测试
- 如果写测试明显不成比例，就运行等价脚本或命令
- 如果两者都不现实，就精确说明未验证的表面

## 验证失败时

当验证失败时：

1. 先识别问题来自本次改动、环境问题，还是既有故障
2. 缩小失败面
3. 只有在存在具体理由时才重试
4. 把既有失败和新回归分开汇报

## 信心用语

用词要精确：

- `verified` 表示已实际检查
- `inferred` 表示基于代码或证据推断
- `not verified` 表示无法直接验证

不要混淆这三类表述。
