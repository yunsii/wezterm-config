当你需要判断某条规则应该放在文档、脚本、hook、skill 还是 plugin 中时，阅读本文件。

## 默认值

如果某种行为必须每次都发生，就不要只依赖提示词记忆。

当一致性重要时，应优先把规则做成实现，而不是停留在说明层。

## 按强度放置规则

文档适合承载：

- 稳定的决策规则
- 路由说明
- 概念边界
- 非确定性判断指导

脚本适合承载：

- 可重复的命令序列
- 环境初始化
- 验证流程
- 维护任务

hook 适合承载：

- 确定性强制约束
- 前置条件检查
- 自动格式化
- 文件保护
- 围绕工具调用的反馈闭环

skill 或 plugin 适合承载：

- 可复用的多步工作流
- 工具编排
- 共享操作能力
- 带环境感知的自动化能力

## 何时下沉

可以问：

- 它是否必须每次发生？
- 是否可以自动检查？
- 是否涉及多个步骤？
- 是否会跨仓库或跨任务复用？

如果答案是肯定的，它通常就不应只停留在文档层。

## 安全性

自动化应当显式、可审查、边界清晰。

优先：

- 清楚的输入和输出
- 可见的失败模式
- 窄作用范围
- 在可行时提供 dry-run 或 preview 路径

避免让操作者感到意外的隐形自动化。
