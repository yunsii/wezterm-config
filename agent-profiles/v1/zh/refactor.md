# Refactor（中文对照）

Source of truth:
[../en/refactor.md](../en/refactor.md)

本文档是英文版的中文对照参考。
若中英文存在差异，以英文版为准。

在重组现有代码、替换子系统或简化复杂逻辑之前，阅读本文件。

## 默认值

不要重写尚未理解的东西。

代码脏乱并不意味着可以跳过分析。

## 先理解

在重构前，应先识别：

- 归属入口点
- 关键数据流
- 状态迁移
- 不变量
- 对外可见行为
- 隐藏依赖
- 可能的回归面

如果这些还不清楚，就继续阅读和建模，而不是直接重写。

## 顺序

推荐顺序：

1. 理解当前行为
2. 提炼不变量
3. 在不改行为的前提下降低偶然复杂度
4. 如有需要，再单独引入行为变化

不要随意把清理和语义重设计混在一起。

## 遗留区域

面对遗留区域时，优先做包裹和收敛，而不是整体替换。

常用手段包括：

- 用更窄的边界包住不稳定逻辑
- 添加表征测试
- 在不改行为的前提下提取 helper
- 先降低分支复杂度，再调整语义

## 风险控制

每一步重构都应有对应的验证路径。

如果系统测试不足，应进一步收缩改动范围。
如果逻辑业务关键，应优先小步且可回退的改动。

## 避免

避免：

- 只是因为某个子系统显得老旧就替换它
- 在缺少验证时同时做重命名和重设计
- 在未理解调用路径前进行大范围文件移动
- 把重命名、移动和行为变化混在一个步骤里
- 仅凭审美驱动的大型重构
